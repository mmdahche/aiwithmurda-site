# Research + Launch System

This system keeps research tied to a decision and launch tied to a verified product.

## Decision research brief


## Evidence rules

- Use current primary sources for software, payment, platform, policy, and technical behavior.
- Use first-party offer pages for market comparisons when possible.
- Place citations next to supported claims.
- Label calculations and inference.
- Do not present a search snippet as complete evidence.
- Preserve the date because AI tools and platform rules change quickly.

## Offer architecture

Define each tier by transformation, not file count:

| Field | Entry | Implementation | Continuity |
| --- | --- | --- | --- |
| Buyer state | New | Ready to install | Already operating |
| Outcome | First result | Full working system | System stays current |
| Access | Self-serve | Permanent baseline | Recurring release channel |
| Support | Documentation | Guided setup | Release notes and migrations |
| Ownership | Included files | Launch edition | Downloads received while active |

For every offer, state:

- Price due today
- Renewal amount and interval
- What remains after cancellation
- What requires active access
- Refund and support expectations
- Exclusions and prerequisites

## Launch proof stack

Before promotion, collect:

1. Product inventory and screenshots
2. Real user path from account to outcome
3. Checkout display and receipt
4. Entitlement and download proof
5. Cancellation and recovery behavior
6. Mobile and desktop verification
7. Buyer FAQ and objection answers
8. Support and incident owner

## Launch content map

Create content from evidence:

- Problem: what the old workflow costs
- Mechanism: how the system changes the work
- Build proof: setup or implementation receipt
- Failure proof: what broke and how it was corrected
- Outcome: what the user can do now
- Offer: exact access, price, renewal, and next action

Do not teach the audience to imitate Murad's public challenge unless that is the product. The public build is proof; the customer assignment must match the paid outcome.

## Prelaunch rehearsal

Test with disposable data:

- New account
- Returning account
- Existing lower-tier buyer
- Mixed one-time and recurring checkout
- Duplicate webhook
- Unpaid or failed session
- Active subscription
- Cancel at period end
- Subscription ended
- Permanent toolkit still available
- Update feed revoked only after period ends
- Customer portal

## Launch dashboard

Track:

- Visitors by source
- Account creation
- Checkout starts
- Paid conversions
- Permanent entitlements
- Active update subscriptions
- Payment failures
- Cancellations
- Support requests
- First setup completion
- Asset downloads
- Time to first verified result

## Launch review

After the first week:

1. Compare objections with actual buyer behavior.
2. Identify where setup stalls.
3. Review refunds, failed payments, and cancellations.
4. Improve the product before increasing traffic.
5. Version every changed asset and communicate material updates.
